## Key Changes

### Meeting Booked Side-effect
When status is set to `meeting_booked` and the "Meeting held" checkbox is checked, `updateProspect` creates an `Interaction` row (type: `meeting`, direction: `outbound`, summary: `[meeting_booked] ...`) and auto-schedules 3 `FollowUp` rows if none exist yet: Follow-up 1 at +4 days, Follow-up 2 at +7 days, Breakup at +11 days (i.e. +4 from Follow-up 2). Guard: checks for existing `[meeting_booked]`-prefixed interactions to prevent duplicate rows on repeat saves.

### Client Paid Side-effect
When the "Client paid" checkbox is checked and a `deal_amount` is provided, `updateProspect` creates a `Deal` row (`status: "paid"`). Guard: checks for an existing paid deal before inserting. UI shows existing deal amount/currency instead of the input fields once a deal exists.

### lastContactAt Update
Updated on any save that has either notes content or creates a meeting interaction. Both code paths assign `newLastContactAt = new Date()` and include it in the `prospect.update` call.

### Excel Export
`GET /dashboard/api/export` — requires auth session. Fetches all prospects with relations and writes 4 sheets:
- **Pipeline** — one row per prospect (status, tier, source, assigned, lastContactAt)
- **Interactions** — all interaction rows flattened with prospect name/id
- **Deals** — all deal rows flattened
- **Follow-ups** — all follow-up rows with due date and completion status

Filename format: `midascreed-pipeline-YYYY-MM-DD.xlsx`.

## Architecture Decisions
- Follow-up scheduling happens inside `updateProspect` rather than a separate endpoint — keeps the UX simple (one Save action triggers everything). The idempotency guard prevents duplicate scheduling.
- Breakup is scheduled at +11 days from meeting (not +4 from now), matching the spec: FU2 is +7, Breakup is +4 after FU2.
- Export uses a Next.js Route Handler (`app/dashboard/api/export/route.ts`) in the `(dashboard)` subtree so it inherits dashboard middleware auth.
