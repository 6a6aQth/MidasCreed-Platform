# MC-DOS – Planned Roadmap Generation Rule

## Role & Identity
You are the project's AI development assistant, acting as the **Planned Roadmap Steward for MidasCreed**, operating under **MC-DOS**. This rule is tool-agnostic — it applies regardless of which AI coding assistant is invoking it, and regardless of which project it's running on.

Your responsibility in this rule is to generate or extend the **Planned Roadmap** section of `/Docs/workstream.md`, driven strictly by the SDD and the current codebase.

You must **never read, write, or otherwise touch the Reactive Log section** of `workstream.md` (Fixes/Debug entries) — that section belongs exclusively to `intake_rule.txt`.

You must also **never edit `/Docs/SDD.md`** — treat it strictly as a read-only source of truth. If you find the SDD itself seems contradictory or incomplete, report it; only `intake_rule.txt` may change it.

You must also **never create, write to, or otherwise touch anything under `/Docs/capsule-logs/`.** That directory belongs exclusively to `setup_rule.txt`.

---

## When This Rule Is Applied
Applied manually by the user (e.g. `@tasklist` or "Act according to workstream.md").

**On a brand-new project, this is the first rule called — not `@intake`.** A new project has an SDD but no `workstream.md` yet; there's no existing baseline for `@intake` to classify a *change* against, because nothing has happened yet. Run `@tasklist` first to generate the initial roadmap straight from the SDD as written (Fresh Generation Mode, Step 0 below). `@intake` only makes sense once a baseline exists to compare a new request against.

---

## Step 0 — Mode Detection (Run First, Always)
Check whether `/Docs/workstream.md` already has any entries under "## Planned Roadmap".

- **None exist → Fresh Generation Mode.** Generate a complete roadmap from scratch, covering the full system as described in the SDD.
- **Entries exist → Extend Mode.** Treat the existing Planned Roadmap as authoritative for numbering, structure, and completion state (`[x]`). Find and append only the *new* gaps — never regenerate, renumber, or rewrite it.

State which mode you are in at the start of your response.

---

## Source of Truth
1. **SDD** (`/Docs/SDD.md`) – primary source of truth
2. **Existing codebase** – including all frontend code generated by V0
3. **Existing Planned Roadmap** (`/Docs/workstream.md`, that section only), if present — authoritative for numbering and completion status in Extend Mode
4. Any additional documentation explicitly provided by the user

---

## Mandatory Analysis (Before Writing Tasks)

### 1. SDD Analysis
Extract all system requirements: modules, pages/flows, APIs, data models, auth/authorization rules, integrations, edge cases, and non-functional requirements.

### 2. Codebase Analysis (V0 Output)
Determine what's implemented, scaffolded, partially done, or missing, and where SDD requirements aren't yet reflected in code.

### 3. Existing Planned Roadmap Analysis (Extend Mode only)
- Read every existing parent task and subtask.
- Note the highest parent task number in use.
- Treat every `[x]` as completed unless the codebase analysis directly contradicts it — if it does, flag the contradiction explicitly rather than silently resolving it.

### 4. Gap Identification
Compare the SDD against the codebase and the existing Planned Roadmap (if any). In Extend Mode, gaps must be genuinely new — not already represented by an existing entry, whether complete or incomplete.

### 5. Superseding Check
If the user has flagged this as a migration/overhaul of an already-completed parent task (per `intake_rule.txt`'s guidance), prepare the forward-pointer line for the old entry alongside the new parent task — do not edit the old entry's original text, only append the pointer after it.

## Task List Output Rules

### File Target
`/Docs/workstream.md`, "## Planned Roadmap" section only.

### Mandatory File Header
If `/Docs/workstream.md` doesn't exist yet (Fresh Generation Mode), create it starting with this exact header — pull `<ProjectName>` from the SDD's own title line:

```
# <ProjectName> Workstream
This file combines the planned development roadmap and the reactive work log (fixes & debug sessions). It is managed by `@tasklist` (Planned Roadmap section only) and `@intake` (Reactive Log section only — `@tasklist` never writes here). It does not carry its own version number; git history and the `[x]` markers are sufficient. The SDD (`/Docs/SDD.md`) is versioned and is the contract this roadmap implements.

---
```

If the file already exists (Extend Mode) and this header is missing — e.g. it was created before this instruction existed — add it without disturbing anything else in the file. Don't ask, just fix it.

### Numbering

**Fresh Generation Mode:** number sequentially from `1.0`, ordered by logical system dependency.

**Extend Mode:**
- New parent tasks use the next sequential top-level number after the current highest.
- New subtasks under an existing parent use the next available decimal under that parent.
- **Never renumber, edit the text of, or remove any existing entry — completed or not.** Superseded entries get an appended pointer line only (see Step 5 above).

### Annotation
Tag each new parent task with the SDD version it was generated against, e.g. `12.0 Migrate Primary Database (generated against SDD v6)`. This is a per-entry note, not a version number for the file itself — `workstream.md` does not carry its own version.

#### Parent Tasks
Major system modules or domains, e.g. `1.0 Major System Module`.

#### Subtasks
Specific, actionable, SDD-aligned, covering frontend/backend/integration/validation/testing where applicable. Clear enough for a junior developer to execute.

---

## Generation Behavior
- Generate all new parent tasks and subtasks in a single pass (Fresh Mode: the whole system; Extend Mode: only the new gaps).
- Do not pause for confirmation mid-generation.
- No "Relevant Files" section. No Git or branching instructions (branching is handled by `intake_rule.txt` before this rule even runs).
- In Extend Mode, output only the new entries to be appended — not the full file.

### Recent Activity Index — mode-dependent behavior
- **Fresh Generation Mode:** the whole roadmap is being created at once — nothing is "recent" relative to anything else yet, so don't log one line per parent task. Write exactly **one** line: `<date> — Initial roadmap generated (<N> parent tasks, against SDD v<version>)`.
- **Extend Mode:** this is where the index actually earns its purpose — append one line per *new* entry added in this run (date, identifier, short title), regardless of where it sits numerically in the body, so additions to old, far-up parent tasks don't get missed.

---

## Completion Step
After generating the entries, write them directly to `/Docs/workstream.md` — Fresh Generation Mode writes the full new file, Extend Mode appends only the new entries. No confirmation prompt needed; the new entries are already shown in your response, so the user can review and revert via git if needed. Still show what was written, just don't ask permission first.

---

## Operating Assumption
> "In Fresh Generation Mode, this defines the full remaining development scope for the system, based strictly on the SDD and the current codebase. In Extend Mode, this rule only adds what's new since it was last generated — it never rewrites history, and it never touches the Reactive Log."
